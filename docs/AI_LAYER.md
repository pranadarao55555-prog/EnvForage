# EnvForge — AI Layer Design

> **Version**: 0.4.0
> **Status**: Phase 4 — In Progress
> **Last Updated**: 2026-05-14

---

## Purpose

The AI Layer provides intelligent troubleshooting, explanation, and repair script
generation. It bridges structured diagnostic data and human-readable guidance.

**Core Principle**: The AI layer is a **reasoning assistant**, NOT a command executor.
It reasons about problems and produces structured suggestions. Actual script generation
always goes through the Template Engine — never raw LLM output.

---

## Responsibilities

| Responsibility | AI Does | AI Does NOT Do |
|---------------|---------|---------------|
| Explain errors | ✓ | Execute anything |
| Suggest fixes | ✓ (structured) | Directly return shell commands |
| Rank fix steps | ✓ | Skip safety filters |
| Generate repair scripts | Suggests intent → Template Engine renders | Write raw scripts |
| Customize templates | Suggests params → Template Engine renders | Modify template files |

## LLM Provider Abstraction

The AI layer uses a pluggable provider interface:

```python
class LLMProvider(ABC):
    @abstractmethod
    async def complete(
        self,
        system_prompt: str,
        user_message: str,
        response_schema: type[BaseModel]
    ) -> BaseModel:
        ...
```

Implementations:
- `OpenRouterProvider` — Multi-model via OpenRouter (✅ Implemented). See [ADR-009](./decisions/ADR-009-openrouter-primary-gateway.md).
- `MockProvider` — For testing; deterministic responses (✅ Implemented)
- `OpenAIProvider` — GPT-4o, GPT-4-turbo (🔲 Planned — use OpenRouter with `openai/gpt-4o` instead)
- `OllamaProvider` — Local inference (Llama 3, Mistral) — no data leaves device (🔲 Planned)

The provider is configured via environment variable: `ENVFORGE_LLM_PROVIDER`
and instantiated via the factory function `get_provider()`.

## Input: Structured Diagnostic Context

The AI receives structured, sanitized context — NOT raw terminal output.

```python
class TroubleshootRequest(BaseModel):
    diagnostic: DiagnosticReport       # Hardware/OS info
    verification: VerificationResult   # Which checks passed/failed
    profile: EnvironmentProfile        # What was being set up
    user_description: str              # Optional: user's description of the issue
    max_words: int = 500               # Response length limit
```

**Why structured input?**
- Prevents prompt injection from malicious diagnostic data
- Ensures AI has clean, typed context
- Makes AI responses more precise and reproducible

## Output: Structured Fix Suggestions

The AI returns a Pydantic model, NOT raw text.

```python
class SuggestedFix(BaseModel):
    step: int
    title: str
    description: str                   # Plain English explanation
    severity: Literal["CRITICAL", "WARNING", "INFO"]
    safe_commands: list[str]           # Read-only diagnostics only (e.g., nvidia-smi)
    repair_template_id: str | None     # If a repair script template exists

class TroubleshootResponse(BaseModel):
    session_id: str
    root_cause: str                    # Plain English root cause analysis
    suggested_fixes: list[SuggestedFix]
    repair_script_available: bool
    confidence: float                  # 0.0–1.0
    disclaimer: str                    # Remind user to review before running
```

## Safety Filter

Before any AI output is exposed to the client, it passes through `SafetyFilter`:

```python
class SafetyFilter:
    BLOCKED_PATTERNS = [
        r"rm\s+-rf",
        r"format\s+[A-Z]:",
        r"dd\s+if=",
        r"DROP\s+DATABASE",
        r"shutdown\s+/s",
        r":(){:|:&};:",   # fork bomb
    ]

    def validate(self, text: str) -> str:
        """Returns sanitized text or raises SafetyViolationError."""
        for pattern in self.BLOCKED_PATTERNS:
            if re.search(pattern, text, re.IGNORECASE):
                raise SafetyViolationError(pattern=pattern, text=text)
        return text
```

`SafetyViolationError` is logged for audit and the response is blocked.

## System Prompt Design

The system prompt enforces safe, structured behavior:

```
You are EnvForge AI, an expert ML/AI environment troubleshooting assistant.

RULES (NON-NEGOTIABLE):
1. You analyze problems — you do NOT execute commands.
2. You return ONLY valid JSON matching the provided schema.
3. You NEVER suggest destructive commands (rm -rf, format, DROP TABLE, etc.).
4. If you are uncertain, say so explicitly in the description field.
5. All suggested shell commands must be READ-ONLY diagnostic commands only
   (e.g., nvidia-smi, python --version, pip show). Never suggest install/uninstall.
6. Repair scripts are generated by EnvForge's template engine, not by you.
   You only suggest which repair_template_id to use.

CONTEXT:
You will receive a structured diagnostic report and verification results.
Analyze these to identify the root cause and suggest ordered remediation steps.
```

## Repair Script Generation Flow

```
AI suggests: repair_template_id = "repair_cuda_upgrade"
                │
                ▼
Backend looks up Template: templates/repair/repair_cuda_upgrade.sh.j2
                │
                ▼
Renders with TemplateContext (AI-influenced params validated by Pydantic)
                │
                ▼
SafetyFilter validates rendered output
                │
                ▼
Repair script returned to user for REVIEW before any execution
```

**Key**: The AI never writes shell script text. It selects template IDs.
The Template Engine produces the actual script, with safety guarantees.

## Rate Limiting (Phase 4.5)

AI endpoints are rate-limited:
- `POST /troubleshoot`: 10 requests/hour per IP (unauthenticated)
- `POST /repair`: 5 requests/hour per IP
- Authenticated users: higher limits (Phase 6)

Implementation: Redis + `slowapi` (FastAPI rate limiting library)

---

## Audit Logging

All AI interactions are logged for safety auditing:
- Input context hash (not PII)
- Safety filter outcomes
- Session ID
- Provider used
- Token usage

Logs are stored in PostgreSQL `ai_audit_log` table.
